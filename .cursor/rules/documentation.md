# Documentation Guidelines

## Core Principle
**Keep documentation minimal and consolidated.**

## What to Create
- ✅ **README.md (root)** - Single source of truth
  - Local setup instructions
  - Concise feature list
  - Simple testing guide
  - Architecture overview (brief)
  - Troubleshooting section

## What NOT to Create
- ❌ Multiple .md files (SETUP.md, TESTING.md, ARCHITECTURE.md, etc.)
- ❌ Implementation summaries
- ❌ Status documents
- ❌ Verbose guides
- ❌ Redundant documentation files

## Exception
Code-level documentation is fine:
- API documentation in comments
- Inline code comments
- JSDoc/TSDoc for complex functions

## Rule
**Before creating ANY .md file, ask:**
1. "Can this go in README.md instead?"
2. "Is this truly necessary, or am I over-documenting?"

**Default answer should be: Add to README.md**
